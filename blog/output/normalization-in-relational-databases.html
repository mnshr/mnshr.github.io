Normalization in Relational Databases

Published: Fri 06 June 2014
By Machined Heuristics

In Blog.

What is Normalization?

  • Normalizing data
  • a concept by Ted Codd, (relational db creator + entity relationship modeling)
  • With normalization all the entities end up as individual objects that ensures that we have a well designed db
  • With no data, we can't normalize | with too much data it is difficult to normalize as data has to be reorganized and change structure
    • We should have a sample of the data that is representative to help us normalize

Normalization terms

  • Entity a group of individual records, identified by primary key (single/multiple columns composite key)
    • Candidate keys are evaluated to create the PK
    • There might be more than 1 unique keys, normalization should work with both, surrogate key & pk
  • Dependency something which could be derived, given another attribute like a pk can be used to derive name, hence name is a dependency

Denormalization

  • In a normalized system, we have multiple tables each representing an entity
  • We could also combine all the tables to fewer so that when we query we have fewer joins and run faster queries
  • This could lead to redundancy, since many columns could be duplicated across tables and a single update will be propogated to multiple tables
  • Normalization is usually done in transactional systems where there are lots of inserts/updates/deletes
  • Denormalization is mostly done for querying/analytical data warehouse systems for efficient querying (having lots of duplication but fast querying)
  • Duplication happens with a Central Fact table and multiple dimensional tables which have extra details (star schema)
  • Data that is fed to normalization process is called Un-normalized (different than denormalized)

Levels of Normalization

Common normalization forms (1st, 2nd, 3rd)

  • Every attribute must provide a fact about a key, the whole key and nothing but the key
  • 1st normal no repeating groups/columns e.g. multiple phone numbers must not be different columns
  • 2nd normal non key columns shouldn't be dependent on part of the primary key (must be checked if we have a composite primary key)
    • We shouldn't be able to derive info about other columns from just 1 column of the composite pk
  • 3rd normal info should be discoverable only through the key (directly, or indirectly through a foreign key)
  • BCNF boyce codd normal form normalization should be able to work with surrogate/candidate key and same rules can be applied using the surrogate key (i.e. even if it isn't the formal primary key)
  • 4th normal multi-valued dependencies
  • 5th normal try to decompose the data as much as possible without any data loss
  • 6th normal and so on, but most db designs look upto 3NF

Overnormalization

  • It may be sometimes good to not follow the rules exactly
  • Choice between having a complete table for the entity or just a couple of columns
    • e.g. address table for every country, state, city, zip code OR just zip code is enough
    • For a customer data spread across the world it is better to create an address table, for just our business office addresses (which are limited) perfect normalization may not be a good idea
